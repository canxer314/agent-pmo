# /close — 项目收尾

项目生命周期的终点。汇总全周期数据，生成决算和复盘。

## 定位

| Skill | 职责 | 输出 |
|-------|------|------|
| `/monitor` | 运营监控 | 健康报告 |
| `/close` | **项目收尾 — 决算 + 复盘** | 决算报告 + 绩效记录 + 复盘 |
| `/query` | 跨项目查询 | 知识复用 |

**典型工作流**: 终验完成 → `/close` 收尾 → 复盘沉淀 → 知识库更新

---

## Behavior

### Step 0: 确定范围

| action | 行为 |
|--------|------|
| `close`（默认）| 完整收尾流程 |
| `retro` | 仅复盘 |

### Step 1: 选择项目

```bash
obsidian search query="type/project"
```

过滤 `status:executing` 或 `status:closing` 的项目，或让用户直接指定。

### Step 2: 汇总项目全周期数据

读取项目全部关键文档：

```bash
obsidian read path="项目库/{project}/00-项目章程.md"
obsidian read path="项目库/{project}/01-合同/主合同关键条款.md"
obsidian read path="项目库/{project}/02-计划/里程碑计划.md"
obsidian read path="项目库/{project}/04-监控/预算执行表.md"
obsidian read path="项目库/{project}/04-监控/风险登记册.md"
obsidian read path="项目库/{project}/03-执行/变更记录/"
```

汇总数据：
- 合同金额 vs 实际收款
- 预算 vs 实际成本
- 计划工期 vs 实际工期
- 变更次数及影响
- 风险发生情况
- 交付物清单

### Step 6: 双提议

```markdown
提议 1 — 知识沉淀建议：
1. 是否将复盘中的经验教训写入知识库？
2. 是否更新客户档案中的合作历史？
3. 是否更新供应商档案中的评价？

提议 2 — 后续行动：
1. 质保期跟踪：建议设置定期提醒
2. 客户关系维护：建议记录下次拜访计划
3. 尾款/质保金：建议跟踪收款节点

请回复编号确认，或"跳过"
```

### Step 7: 更新项目状态

```bash
obsidian property:set path="项目库/{project}/00-项目章程.md" name="tags" value="type/project,status/closed,priority/p0" type=list
```

更新 MOC-Projects：

```bash
obsidian read path="MOC-Projects.md"
```

将项目状态更新为 closed。

## 与其他 Skill 的协作

| 场景 | 组合 |
|------|------|
| 收尾前检查健康度 | `/close` 前 → `/monitor` |
| 复盘后查询历史 | `/close` 后 → `/query "同类项目复盘"` |
| 客户档案更新 | `/close` → 客户档案 |

---

## Notes

- `/close` 是项目生命周期的**终点**，状态更新为 closed 后不可随意更改
- 决算数据必须准确，具有财务审计价值
- 复盘应诚实记录失误，不粉饰
- 经验教训的沉淀是 `/close` 最重要的价值
