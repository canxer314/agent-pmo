# /meeting — 会议纪要

结构化记录会议内容，提取决议项和待跟踪事项。

## 定位

| Skill | 职责 | 输出 |
|-------|------|------|
| `/meeting` | **会议纪要 — 项目沟通留痕的唯一通道** | `03-执行/会议纪要/` 下的会议文档 |
| `/change` | 变更管理 | 变更记录 |
| `/monitor` | 运营监控 | 跟踪决议项闭环 |

**典型工作流**: 开会 → `/meeting` 记录 → `/monitor` 跟踪决议项 → `/change`（如涉及变更）

---

## Behavior

### Step 1: 确定会议阶段和类型

#### 1a. 判断会议阶段

根据用户输入或上下文判断会议属于哪个阶段：

| 阶段 | 触发关键词 | 存储位置 | 关联 Skill |
|------|-----------|----------|------------|
| 售前技术交流 | '技术交流'、'售前'、'方案汇报'、'客户交流' | `投标档案/{bid}/技术交流记录/` | `/bid` |
| 执行阶段会议 | '周例会'、'启动会'、'评审会'、'变更会'、'项目会' | `项目库/{project}/03-执行/会议纪要/` | `/initiate`、`/change`、`/monitor` |

**判断逻辑**：
- 用户提到投标、售前、技术交流 → 售前阶段
- 用户提到项目编号（PJ-xxx）或已有项目名 → 执行阶段
- 未明确时，询问用户："这是售前技术交流还是执行阶段的项目会议？"

#### 1b. 售前技术交流（type=tech-exchange）

```bash
# 如提供 bid 参数，读取投标档案上下文
obsidian read path="投标档案/{bid}/客户需求分析.md"
obsidian read path="投标档案/{bid}/解决方案.md"
```

收集信息：

| 字段 | 说明 |
|------|------|
| 会议日期 | 默认今天 |
| 交流主题 | 如"方案汇报v2" |
| 参会人 | 双方参会人员 |
| 交流要点 | 客户关注点、技术问题、反馈 |
| 客户反馈 | 对材料的评价、进一步需求 |

收集完基本信息后，询问：

```
本次是否为 PoC/演示？
→ 是（将额外记录 PoC 环境、测试用例和结果）
→ 否（使用标准技术交流记录模板）
```

#### 1c. 执行阶段会议

```bash
obsidian search query="type/project"
```

列出活跃项目供选择。

收集信息：

| 字段 | 说明 |
|------|------|
| 会议日期 | 默认今天 |
| 会议类型 | 周例会 / 启动会 / 评审会 / 变更会 / 专题会 |
| 参会人 | 双方参会人员 |
| 会议主题 | 一句话 |

### Step 3: 双提议

#### 售前技术交流的双提议

```markdown
✓ 确定性链接已自动建立：
- [[解决方案]]（如存在）— 本次交流的方案基础
- [[客户需求分析]]（如存在）— 本次交流的需求上下文
{如为 PoC} ✓ 确定性链接：PoC 记录已自动链接 [[解决方案]] 和 [[客户需求分析]]

提议 1 — 关联建议：
1. [[投标档案/{bid}/客户需求分析]] — 如有新增需求更新
2. [[投标档案/{bid}/解决方案]] — 如有方案调整更新
3. [[投标档案/{bid}/售前材料清单]] — 材料版本记录

提议 2 — 后续行动建议：
1. 本次交流的客户反馈是否需要更新到「客户需求分析」？
2. 是否需要更新「解决方案」版本记录？
3. 是否需要更新「售前材料清单」中的材料版本？
{如为 PoC}
4. PoC 结果是否需要更新 [[解决方案]] 的「风险与应对」或「产品/服务配置」？
5. PoC 暴露的问题是否值得沉淀到 [[知识库/经验教训/]]？

🏷️ 投标状态建议：→ tech-exchange-loop

提议 3 — 知识沉淀：
1. 本次交流中是否有值得沉淀为标准回应的技术问题或异议处理？
   → 如确认，创建 知识库/技术答疑/{主题}.md（自动链接来源技术交流记录）
2. 该回应是否适用于其他活跃投标？{如有适用投标} 建议跨投标链接 → 列出候选投标清单

请回复编号确认，或纠正状态，或"跳过"
```

---

## Notes

- `/meeting` 是项目沟通的**唯一留痕通道**
- **会议类型决定存储位置**：
  - 售前技术交流 → 投标档案/技术交流记录/（关联 `/bid`）
  - 执行阶段会议 → 项目库/03-执行/会议纪要/（关联 `/initiate`、`/change`、`/monitor`）
- 决议项必须有责任人 + 截止日期，否则无法跟踪
- 会议纪要的"风险/变更识别"section 是连接 `/monitor` 和 `/change` 的桥梁
- 建议每次会议都检查上次会议的决议项闭环情况
- 售前技术交流记录客户反馈，用于迭代解决方案和材料版本
