# /prospect — 线索管理

记录售前线索，跟踪客户接触全过程。

## 定位

| Skill | 职责 | 输出 |
|-------|------|------|
| `/prospect` | 线索管理 — 记录和跟踪售前线索 | `线索池/` 下的线索文档 |
| `/bid` | 投标管理 — 投标全过程留痕 | `投标档案/` 下的投标文件夹 |
| `/initiate` | 项目立项 — 中标后创建项目结构 | `项目库/` 下的项目文件夹 |

**典型工作流**: `/prospect` 记录线索 → 跟进更新 → `/bid` 投标 → `/initiate` 立项

---

## Behavior

### Step 0: 确定动作

根据 `action` 参数：

| action | 行为 |
|--------|------|
| `new`（默认）| 创建新线索 |
| `update` | 更新现有线索 |
| `list` | 列出所有线索 |

### Step 1: 创建新线索（action=new）

#### 1a. 收集信息

与用户对话收集（未提供的字段会询问）：

| 字段 | 说明 | 示例 |
|------|------|------|
| 客户名称 | 客户全称 | 某市人民政府 |
| 需求主题 | 一句话描述需求 | 政务数字化平台升级 |
| 需求概述 | 客户要做什么、为什么做 | [详细描述] |
| 预计金额 | 预估合同额 | 500万 |
| 预计签约时间 | 客户预期或销售判断 | 2026-Q3 |
| 当前阶段 | 初步接触/需求确认/方案沟通/商务谈判 | 需求确认 |
| 竞争对手 | 已知的竞争对手 | 某科技公司、某软件公司 |
| 关键干系人 | 客户方决策链 | 王主任（信息中心主任） |
| 信息来源 | 怎么知道这个线索的 | 老客户转介绍 |

#### 1b. 检查客户档案

```bash
obsidian search query="客户档案-{client}"
```

- 找到 → 记录客户档案路径，后续双提议时建议关联
- 未找到 → 双提议时建议新建客户档案

#### 1c. 准备线索文档内容

文件路径：`线索池/线索-{客户}-{主题}.md`（待用户确认后写入）

```markdown
---
title: "线索-{客户}-{主题}"
type: prospect
stage: 初步接触 | 需求确认 | 方案沟通 | 商务谈判
estimated_value: 5000000
expected_date: 2026-Q3
owner: 业务经理
client: "{客户}"
source: 老客户转介绍
competitors:
  - 某科技公司
  - 某软件公司
stakeholders:
  - 王主任（信息中心主任）
created: YYYY-MM-DD
updated: YYYY-MM-DD
tags:
  - type/prospect
  - stage/需求确认
---

# 线索：{客户} — {主题}

> 预计金额：500万 | 预计签约：2026-Q3 | 阶段：需求确认

## 需求概述

[客户要做什么、为什么做、什么时间要]

## 客户背景

[客户行业、规模、信息化现状]

## 竞争态势

| 竞争对手 | 已知优势 | 已知劣势 |
|----------|----------|----------|
| 某科技公司 | [优势] | [劣势] |

## 干系人

- 王主任（信息中心主任）— [角色描述]

## 跟进记录

### YYYY-MM-DD

[本次沟通内容、客户反馈、下一步行动]

## 下一步

- [ ] 行动项1
- [ ] 行动项2
```

#### 1d. 双提议

```markdown
提议 1 — 关联建议：
1. [[客户档案-{客户}]] — 客户档案{已存在/不存在}，建议关联
2. [[知识库/经验教训/政务项目-经验教训]] — 历史有同类项目经验，建议关联

提议 2 — 后续行动建议：
1. 是否立即创建客户档案？（如不存在）
2. 是否进入 /bid 投标流程？（如阶段已到方案沟通）

请回复编号确认，或"跳过"
```

#### 1e. 写入线索文档（如用户确认）

```bash
obsidian create path="线索池/线索-{客户}-{主题}.md" content="..."
```

### Step 2: 更新现有线索（action=update）

#### 2a. 搜索线索

```bash
obsidian search query="type/prospect"
```

列出匹配线索，让用户选择：

```
找到以下线索：
1. 线索-某市政府-数字化
2. 线索-某国企-ERP升级

请回复编号选择，或提供线索名称
```

#### 2b. 读取并追加

```bash
obsidian read path="线索池/线索-{客户}-{主题}.md"
```

在文档末尾追加新的跟进记录：

```bash
obsidian append path="线索池/线索-{客户}-{主题}.md" content="

### YYYY-MM-DD

[本次更新内容]

## 下一步

- [ ] 新行动项
"
```

更新 frontmatter 中的 `updated` 和 `stage`（如阶段推进）。

#### 2c. 阶段推进检查

如果 `stage` 从 "商务谈判" 变为 "中标"：
- 提示用户："线索已中标，是否立即走 /initiate 立项？"
- 用户确认后，move 线索到 `线索池/已转化/`

如果 `stage` 变为 "未中标"：
- 提示用户："是否记录未中标原因到知识库/经验教训/？"
- 用户确认后，创建经验教训文档

### Step 3: 列出所有线索（action=list）

```bash
obsidian search query="type/prospect"
```

按阶段分组展示：

```markdown
## 线索总览

### 商务谈判（{N}条）
| 客户 | 主题 | 预计金额 | 预计签约 |
|------|------|----------|----------|
| 某市政府 | 数字化 | 500万 | 2026-Q3 |

### 方案沟通（{N}条）
...

### 需求确认（{N}条）
...

### 初步接触（{N}条）
...
```

## 与其他 Skill 的协作

| 场景 | 组合 |
|------|------|
| 线索中标，需立项 | `/prospect` → `/initiate` |
| 线索进入投标阶段 | `/prospect` → `/bid` |
| 线索未中标，沉淀经验 | `/prospect` → 知识库经验教训 |

---

## Notes

- `/prospect` 是 vault 线索层的**唯一写入通道**
- 线索文档的 `stage` 字段是核心状态，必须准确
- 竞争对手信息可为 `/bid` 的投标策略提供参考
- 干系人信息为项目立项后的干系人管理提供基础
