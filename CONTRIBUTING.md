# Contributing to Knowledge MEMO

## 核心原则：Fork, Don't Consume

Knowledge MEMO 不是一个通用产品。本仓库是针对 **售前 + 解决方案 + 项目管理** 场景定制的项目全生命周期运营系统。

我们把它开源，**不是让你当 SaaS 产品用，是让你看到一个真实跑通的样本**，然后去长出**你自己的那一个**。

不同的项目经理有不同的管理风格，不同的公司有不同的审批流程，不同的行业有不同的验收标准。这套系统在售前-交付一体化场景下跑通了——你的场景可能不同，但结构可以借鉴。

---

## 建议的 Fork 流程

1. **Clone 到本地**
   ```bash
   git clone https://github.com/canxer314/agent-pmo.git
   cd agent-pmo
   ```

2. **用你自己的项目类型替换示例**
   `templates/cards/` 里展示的项目管理模板。你的行业可能完全不同——政务信息化、企业 SaaS、系统集成、纯咨询服务——删掉不适用的，保留可借鉴的结构。

3. **改写 `SCHEMA.md` 的 Card 类型**
   去掉你不需要的，加上你需要的。你的业务可能需要 `type/subcontract`（分包管理）、`type/compliance`（合规审查）等独有的 Card 类型。

4. **改写 `templates/cards/` 下的模板**
   符合你的 frontmatter 偏好、你的标签体系、你的文档写作习惯。

5. **跑一个完整的项目周期，观察哪里不顺手，迭代**
   项目治理不是"装好就好"，是"持续调优"。每次调整都要记录理由和日期。

---

## 如果你想回流贡献

- **Bug fix / 文档改进**：直接 PR 到 main
- **新 Skill 或新 Card 类型**：先开 issue 讨论，确保与现有架构一致
- **哲学立场变更**：不接受（这是本 fork 的设计立场，不是民主投票）

---

## 不接受的 PR

这些不是"看不上"，是和本仓库的灵魂冲突：

- **"auto-ingest" / "scheduled scraping" 等全自动化信息流**
  违背"人必须在 loop 里"原则。项目治理的第一条纪律是：AI 执行，人拍板。自动化决策绕过了项目经理的判断力。

- **删除双提议机制或给 Skill 加 "auto-confirm" 模式**
  双提议机制是刻意的摩擦。它确保每一项写入 vault 的决策都经过人的确认。绕开它就失去了"人在 loop"的强制点。

- **把定制化模板改成"通用示例"**
  失去真实性。模板的价值恰恰在于它是真实跑通过的——有取舍、有行业特征、有个人风格。一旦变成通用示例就平庸了。

- **删除审计可追溯性要求**
  项目数据的修改留痕（日期、原因、前后对比）具有审计价值。这是底线，不可妥协。

---

## 我们特别欢迎的贡献

- **翻译**：README / SCHEMA / AGENTS 的多语言版本
- **Skill 改进**：提升现有 15 个 Skill 的 prompt 质量、修 bug、补 edge case 处理
- **行业适配模板**：针对特定行业（政务信息化、医疗 IT、智能制造等）的 Card 模板和命名规范
- **Platform 兼容**：让 Skills 在非 Claude Code 环境也能运行（Codex、本地 LLM 等）
- **Fork 案例分享**：你改造后的版本——即使不 PR 回来，把 link 发 issue 让我们看看，会帮助其他人想象可能性

---

## 关于"作者"和"用户"

本仓库是单人使用设计。文档中的"你"指的是项目经理本人——一个同时承担售前方案、交付管理、客户关系的人。AI 是助手，不是协作者。

你 fork 之后，"你"就是你。把 SCHEMA 里的状态流转改成你公司的流程，把模板改成你行业的标准——这个 repo 就变成了你自己的项目治理样板。
