# /work-item — 专项工作管理

跨数天、多步骤、有明确产出的重量级临时事项的独立管理通道。与 `/monitor action=track` 的分工：`/work-item` 管理**工作文档内容**（创建、结构组织、产出记录、完成），`/monitor track` 管理**看板中的轻量状态**（todo/doing/done 切换）。

## 设计原则：最小核心 + 按需生长

项目履约中的专项工作五花八门——写汇报材料、排查故障、组织评审会、紧急采购、客户投诉处理、数据迁移——固定模板反而碍事。因此 `/work-item` 采用**柔性结构**：

- **必选核心**（4 个 section）：任何专项工作都需要的骨架
- **可选 section 目录**（10+ 个）：AI 在创建/更新时根据工作性质推荐启用
- **按需生长**：工作推进过程中，用户可以随时说"加一个审阅区"或"不用大纲了"

---

## 可选 Section 目录

| Section | 适用场景 | 典型工作 |
|---------|---------|---------|
| `素材与参考` | 需要收集整理参考资料的 | 汇报材料、方案文档、合同起草 |
| `大纲/逻辑主线` | 产出物有结构骨架的 | 汇报PPT、培训课件、技术方案 |
| `审阅记录` | 有内部或客户审阅流程的 | 汇报材料、交付物、对外文档 |
| `现象/问题描述` | 从异常现象出发排查的 | 故障排查、性能问题、Bug 修复 |
| `排查/诊断过程` | 需要记录诊断步骤的 | 服务器故障、数据异常、接口报错 |
| `根因与方案` | 需要做根因分析的 | 生产事故、客户投诉、质量缺陷 |
| `议程/流程` | 需要组织多人参与活动的 | 客户拜访、评审会、培训、验收会 |
| `参与方/资源` | 涉及多方协调的 | 跨团队协作、供应商对接、客户沟通 |
| `决策记录` | 需要多轮决策或审批的 | 采购选型、方案比选、预算追加 |
| `审批节点` | 有正式审批流程的 | 合同变更、预算申请、采购订单 |
| `执行日志` | 需要记录日常进展的 | 所有类型（默认启用） |
| `数据/附件` | 工作过程产生数据或附件的 | 数据迁移、测试报告、配置快照 |

AI 在创建和更新时主动判断是否需要这些 section，向用户提议。用户可以随时增删。

## Behavior

### Step 0: 确定动作

| action | 行为 |
|--------|------|
| `new`（默认）| 创建专项工作文档 + 在看板创建 TMP 行 |
| `update` | 更新工作文档内容（推进步骤、补充内容、调整结构） |
| `close` | 完成工作，填写完成记录，触发看板归档 |

---

### Step 1: 创建专项工作（action=new）

#### 1a. 选择项目

如提供 `project` 参数，直接使用。否则：

```bash
obsidian search query="type/project"
```

过滤 `status:executing` 或 `status:monitoring` 的项目，列出供选择。

#### 1b. 确定来源

| source | 说明 | 后续行为 |
|--------|------|----------|
| `meeting` | 会议决议产生的任务 | 提示用户提供会议纪要路径和决议编号 |
| `standup` | 站会中发现的新任务 | 无需额外上下文 |
| `adhoc` | 领导/客户临时交待 | 提示用户提供来源说明 |

如 `source=meeting`，列出项目下的会议纪要供选择。用户选择后读取决议项。如果选中的决议项已有对应的专项工作文档（通过 wikilink 引用），提示用户避免重复创建。

#### 1c. 收集核心信息

与用户对话收集：

| 字段 | 说明 | 示例 |
|------|------|------|
| 工作标题 | 简洁描述 | 客户年度汇报 / 生产环境CPU故障排查 / Q3采购选型 |
| 负责人 | | 张三 |
| 截止日期 | | 2026-06-15 |
| 产出描述 | 完成后交付什么 | 年度运维报告 PPT + 现场汇报 / 故障分析报告 + 根因修复 |

`source=meeting` 时，从会议纪要的决议项预填。`source=adhoc` 时，额外收集来源说明。

#### 1d. AI 建议 Section 结构

AI 根据工作标题和产出描述，分析工作性质，推荐可选 section：

```
根据你的描述，这项工作建议启用以下 section：

必选（自动包含）：
  ✓ 背景
  ✓ 产出要求
  ✓ 工作步骤
  ✓ 完成记录

建议启用的可选 section：
  1. 素材与参考 — 汇报材料通常需要收集历史数据、上次汇报PPT等
  2. 大纲/逻辑主线 — 汇报类工作的核心，确定叙事结构
  3. 审阅记录 — 汇报材料通常需要内部审阅再对外
  4. 执行日志 — 记录每日推进情况

确认这些 section？可以回复：
  → "全要" 或 "确认"
  → "去掉审阅记录"
  → "加上数据/附件"
  → 直接描述你想要的
```

**分析原则**：AI 根据用户的原话判断，不是让用户从菜单选。几类典型信号：

| 用户说的关键信号 | 推荐 section |
|-----------------|-------------|
| 汇报/PPT/方案/文档/报告/标书 | 素材与参考、大纲/逻辑主线、审阅记录 |
| 故障/报错/异常/挂了/慢了/排查 | 现象/问题描述、排查/诊断过程、根因与方案 |
| 拜访/会议/评审/培训/验收/组织 | 议程/流程、参与方/资源 |
| 采购/选型/招标/比价/审批 | 决策记录、审批节点 |
| 迁移/上线/部署/配置/升级 | 执行日志、数据/附件 |
| 投诉/事故/缺陷/质量问题 | 现象/问题描述、根因与方案、决策记录 |

这些只是推荐起点，用户可以用自然语言调整（"我需要加一个审批区"、"把大纲换成素材区"）。

#### 1f. 在看板同步 TMP 行

```bash
obsidian read path="项目库/{project}/03-执行/交付看板.md"
```

如看板不存在，先创建（使用看板模板）。然后在临时事项区追加：

```
TMP-{next} [[专项工作/{日期}-{标题}|{标题}]] {负责人} todo → {deadline}
```

更新看板 frontmatter 的 `total_active` 计数。

#### 1g. 确定性链接

| 链接 | 条件 |
|------|------|
| `[[03-执行/交付看板]]` | 无条件（同项目固定文件名） |
| `[[03-执行/会议纪要/{日期}-{类型}]]` | `source=meeting` 且来源会议已指定 |

#### 1h. 双提议

```markdown

- [[03-执行/交付看板]] — 已关联（看板 TMP 行已同步）

1. 该专项工作是否与某个里程碑相关？是否需要关联 [[02-计划/里程碑-xxx]]？
```

### Step 2: 更新专项工作（action=update）

#### 2a. 选择工作文档

```bash
obsidian files folder="项目库/{project}/03-执行/专项工作"
```

列出专项工作文档供选择。

#### 2b. 读取并展示当前状态

```bash
obsidian read path="项目库/{project}/03-执行/专项工作/{doc}.md"
```

展示工作步骤 checklist 和最新内容摘要，让用户选择更新区域：

```
当前工作：客户年度汇报
已启用 section：背景 / 产出要求 / 素材与参考 / 大纲 / 审阅记录 / 工作步骤
截止：06-15（还剩 28 天）

更新哪部分？
  1. 推进工作步骤（勾选完成项）
  2. 补充某个 section 的内容（请说明哪个 section）
  3. 添加新的可选 section（如"加一个审批区"）
  4. 修改截止日期
  5. 修改产出描述
  6. 修改负责人
  7. 追加执行日志
```

#### 2c. 动态增减 Section

更新过程中，AI 可以主动建议：

```
这份汇报材料已经迭代了3版审阅，建议添加「决策记录」section 记录每次审阅后的关键决策。
需要我加上吗？
```

用户也可以主动要求：
- "加一个数据/附件区"
- "去掉审阅记录，已经不需要了"
- "把大纲和素材合并"

AI 执行 section 的增减操作：添加时按模板插入对应 section，删除时移除该 section 及其内容（保留到备注区以防丢失）。

#### 2d. 执行更新

根据用户选择更新对应内容。如修改截止日期超过原 deadline 30 天以上，提示是否影响关联里程碑。

#### 2e. 双提议

```markdown
提议：
1. 修改了截止日期，是否需要更新 [[03-执行/交付看板]] 中的 TMP 行日期？
2. 进展是否需要在下次会议中同步？
{如截止日期大幅延期}
3. ⚠️ 截止日期从 {old} → {new}（+{N}天），是否需要通知相关方？

请回复编号确认，或"跳过"
```

```markdown
{如 source=meeting}
3. 是否回写 [[{会议纪要}]]，标记决议 D{N} 已闭环？

提议 2 — 后续行动：
1. 是否触发了新的工作？（如有，创建新的专项工作）

请回复编号确认，或"跳过"
```

#### 3f. 写入

更新工作文档和看板。

## 与其他 Skill 的协作

| 场景 | 组合 |
|------|------|
| 会议决议转为专项工作 | `/meeting` → `/work-item action=new source=meeting` |
| 站会中发现新任务 | `/monitor action=track` 识别 → `/work-item action=new source=standup` |
| 领导临时交待 | 用户直接 → `/work-item action=new source=adhoc` |
| 日常状态切换 | `/monitor action=track` 更新 TMP 行状态 |
| 工作文档内容更新 | `/work-item action=update` |
| 完成归档 | `/work-item action=close` |

---

## Notes

- `/work-item` 管理**工作文档内容**，`/monitor track` 管理**看板轻量状态**
- 结构是生长出来的，不是一次性定死的 —— 开始时少几个 section 完全正常
- 三种来源：`meeting`（会议决议）、`standup`（站会发现）、`adhoc`（临时交待）
- 创建时自动在看板同步 TMP 行，完成时自动归档
- `source=meeting` 时，`source_doc` 应指向具体的会议纪要文档
- 截止日期大幅变更时应提示是否影响关联里程碑
- 完成后的经验沉淀是知识库积累的重要来源
