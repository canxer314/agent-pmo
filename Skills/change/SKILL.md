# /change — 变更管理

记录变更，评估影响，跟踪审批状态。

## 定位

| Skill | 职责 | 输出 |
|-------|------|------|
| `/meeting` | 会议纪要 | 会议文档 |
| `/change` | **变更管理 — 唯一创建变更记录的通道** | 变更记录 CR |
| `/monitor` | 运营监控 | 变更影响跟踪 |

**典型工作流**: `/meeting` 识别变更需求 → `/change` 记录并评估 → `/monitor` 跟踪闭环

---

## Behavior

### Step 1: 确定项目与来源

#### 1a. 选择项目

```bash
obsidian search query="type/project"
```

列出活跃项目供选择。

#### 1b. 读取相关上下文

如 `source=meeting`：

```bash
obsidian read path="项目库/{project}/03-执行/会议纪要/{最近日期}.md"
```

提取会议中识别的变更需求。

### Step 2: 收集变更信息

与用户对话收集：

| 字段 | 说明 |
|------|------|
| 变更内容 | 具体变更描述 |
| 变更原因 | 为什么要变更 |
| 提出人 | 谁提出的 |
| 提出日期 | 默认今天 |

### Step 3: 影响评估（核心）

这是变更管理的核心。必须评估四个维度：

```markdown
## 影响评估

### 工期影响

- 影响天数：+{N} 天
- 受影响里程碑：[[里程碑-xxx]]、[[里程碑-yyy]]
- 关键路径是否受影响：是 / 否

### 成本影响

- 追加成本：+{N} 万
- 成本类别：人力 / 采购 / 其他
- 是否超预算：是 / 否

### 质量影响

- 对交付物质量的影响：
- 是否需要调整验收标准：

### 合同影响

- 是否涉及合同条款变更：
- 是否触发违约条款：
- 客户是否同意书面确认：
```

### Step 4: 生成变更记录

#### 4a. 分配变更编号

项目内顺序：`CR-{NNN}`

搜索已有变更：

```bash
obsidian files folder="项目库/{project}/03-执行/变更记录"
```

确定下一个编号。

#### 4b. 变更记录文档

文件路径：`项目库/{project}/03-执行/变更记录/CR-{编号}.md`

```markdown
---
title: "变更请求 CR-{编号}"
type: change
project: "{project_id}"
cr_id: "CR-{编号}"
change_type: 范围变更 | 需求变更 | 工期调整 | 成本追加 | 其他
proposed_by: "{提出人}"
proposed_date: YYYY-MM-DD
impact_cost: +300000
impact_schedule: +15
approval_status: draft | submitted | under-review | approved | rejected | implemented
tags:
  - type/change
  - status/draft
---

# 变更请求 CR-{编号}

> 项目：[[00-项目章程]] | 提出人：{提出人} | 日期：YYYY-MM-DD

## 变更内容

[具体变更描述]

## 变更原因

[为什么需要变更]

## 影响评估

### 工期影响

- 影响天数：+{N} 天
- 受影响里程碑：
  - [[里程碑-xxx]] — 计划从 {原日期} 调整为 {新日期}
  - [[里程碑-yyy]] — 计划从 {原日期} 调整为 {新日期}
- 关键路径影响：是 / 否

### 成本影响

- 追加成本：+{N} 万
- 成本类别：
- 新预算总额：{原预算} + {追加} = {新总额}
- 是否超原预算：是 / 否

### 质量影响

[对交付物质量的影响]

### 合同影响

- 是否涉及合同条款变更：
- 客户书面确认：已获取 / 待获取 / 不需要

## 审批记录

| 审批人 | 意见 | 日期 | 状态 |
|--------|------|------|------|
| | | | |

## 实施状态

- [ ] 已批准
- [ ] 已实施
- [ ] 已验证

## 关联文档

- 来源：[[03-执行/会议纪要/{日期}-{type}]]
- 受影响：[[02-计划/里程碑计划]]、[[04-监控/预算执行表]]
```

### Step 5: 双提议

```markdown
✓ 确定性链接已自动建立：
- [[00-项目章程]] — 已自动关联
- [[04-监控/预算执行表]] — 已自动关联
- 受影响里程碑 — 已自动关联
- {如 source=meeting} [[03-执行/会议纪要/{date}-{type}]] — 来源会议已自动关联

提议 1 — 关联建议：

提议 2 — 后续行动建议：
1. 是否立即更新里程碑计划？（工期受影响时）
2. 是否立即更新预算执行表？（成本受影响时）
3. 是否需要召开变更评审会？（重大变更时）
4. 是否需要与客户确认书面变更单？

提议 3 — 看板同步（变更批准后）：
1. 本次变更是否产出了新的执行工作？
   → 如确认，在交付看板中创建 REQ 条目，来源标记 CR-{N}
   → CR 文档中记录产出的 REQ 编号，形成双向关联

请回复编号确认，或"跳过"
```

### Step 6: 写入与跟踪

```bash
obsidian create path="项目库/{project}/03-执行/变更记录/CR-{编号}.md" content="..."
```

#### 6a. 更新项目章程变更计数

```bash
obsidian read path="项目库/{project}/00-项目章程.md"
```

在章程中记录变更次数。

#### 6b. 里程碑计划更新（如用户确认）

```bash
obsidian read path="项目库/{project}/02-计划/里程碑计划.md"
```

更新受影响里程碑的计划日期。

#### 6c. 预算执行表更新（如用户确认）

```bash
obsidian read path="项目库/{project}/04-监控/预算执行表.md"
```

追加支出或调整预算。

### Step 7: 输出确认

```markdown
---
变更记录已创建：CR-{编号}

📄 文件：项目库/{project}/03-执行/变更记录/CR-{编号}.md

📊 影响评估：
- 工期：+{N} 天
- 成本：+{N} 万
- 受影响里程碑：{N} 个

{如已更新}
✓ 已更新：里程碑计划
✓ 已更新：预算执行表

下一步：
→ 获取审批后更新变更状态
→ /monitor 跟踪变更实施闭环
```

## 与其他 Skill 的协作

| 场景 | 组合 |
|------|------|
| 会议产生变更 | `/meeting` → `/change` |
| 变更后监控 | `/change` → `/monitor` |
| 重大变更需开会 | `/change` → `/meeting`（变更评审会） |
| 变更影响合同条款 | `/change` → `/contract action=amend` |

---

## Notes

- `/change` 是 vault **唯一创建变更记录的通道**
- 影响评估必须完整，不能遗漏工期/成本/质量/合同任何维度
- 变更审批状态必须及时更新
- 所有变更必须有书面留痕（会议纪要或变更单）
- 变更涉及合同金额、工期或服务范围时，建议走 `/contract action=amend` 创建补充协议
