# /meeting — 会议纪要

结构化记录执行阶段会议内容，提取决议项和待跟踪事项。

## 定位

| Skill | 职责 | 输出 |
|-------|------|------|
| `/meeting` | **会议纪要 — 执行阶段项目沟通留痕的唯一通道** | `03-执行/会议纪要/` 下的会议文档 |
| `/change` | 变更管理 | 变更记录 |
| `/monitor` | 运营监控 | 跟踪决议项闭环 |

> 售前技术交流已移至 `/presales action=tech-exchange`。

**典型工作流**: 开会 → `/meeting` 记录 → `/monitor` 跟踪决议项 → `/change`（如涉及变更）

---

## Behavior

### Step 1: 选择项目与收集信息

#### 1a. 选择项目

```bash
obsidian search query="type/project"
```

列出活跃项目供选择。

#### 1b. 收集会议信息

| 字段 | 说明 |
|------|------|
| 会议日期 | 默认今天 |
| 会议类型 | 周例会 / 启动会 / 评审会 / 变更会 / 专题会 |
| 参会人 | 双方参会人员 |
| 会议主题 | 一句话 |

### Step 2: 读取项目上下文

```bash
obsidian read path="项目库/{project}/00-项目章程.md"
obsidian read path="项目库/{project}/03-执行/交付看板.md"
obsidian read path="项目库/{project}/04-监控/风险登记册.md"
```

了解项目当前状态，帮助判断会议内容的影响。交付看板用于在纪要中嵌入当前执行状态快照。

### Step 3: 结构化记录

文件路径：`项目库/{project}/03-执行/会议纪要/{日期}-{类型}.md`

```markdown
---
title: "会议纪要-{日期}-{类型}"
type: meeting
project: "{project_id}"
meeting_date: YYYY-MM-DD
type: weekly | kickoff | review | change | ad-hoc
participants:
  - 我方：张三、李四
  - 客户：王主任、赵工
tags:
  - type/meeting
---
```

### Step 4: 双提议

```markdown
✓ 确定性链接已自动建立：
- [[00-项目章程]] — 已自动关联
{如 type=kickoff} - [[02-计划/里程碑计划]] — 已自动关联（启动会确认计划）

提议 1 — 关联建议：
1. [[04-监控/风险登记册]] — 如涉及风险识别，建议更新
2. `03-执行/变更记录/` — 如涉及变更需求，建议创建变更记录

提议 2 — 后续行动建议：
1. 决议项是否需要加入个人工作台待办？
2. 识别的风险是否需要录入风险登记册？
3. 变更需求是否需要走 /change 流程？

提议 3 — 专项工作创建：
1. 决议中是否有需要创建专项工作文档的事项？
   → 如有，使用 /work-item action=new source=meeting 逐项创建
   → 会议纪要路径和决议编号将自动传递

请回复编号确认，或"跳过"
```

### Step 5: 写入与跟踪

```bash
obsidian create path="项目库/{project}/03-执行/会议纪要/{日期}-{类型}.md" content="..."
```

#### 5a. 决议项跟踪（如用户确认）

将决议项追加到个人工作台：

```bash
obsidian read path="个人工作台/今日关注.md"
obsidian append path="个人工作台/今日关注.md" content="
## 待跟踪决议（来自 {日期} 会议纪要）

- [ ] D01：[内容]（责任人：{name}，截止：{date}）
- [ ] D02：[内容]（责任人：{name}，截止：{date}）
"
```

#### 5b. 风险登记册更新（如用户确认）

```bash
obsidian read path="项目库/{project}/04-监控/风险登记册.md"
obsidian append path="项目库/{project}/04-监控/风险登记册.md" content="
| R0N | [风险描述] | 高/中/低 | 高/中/低 | 高/中/低 | [措施] | [责任人] | 识别 |
"
```

#### 5c. 触发变更（如用户确认）

提示用户：
```
本次会议涉及变更需求，是否立即走 /change？
→ 说 "change" 或 "变更" 触发 /change
→ 说 "不用" 跳过
```

### Step 6: 输出确认

```markdown
---
会议纪要已记录：{日期}-{类型}

📄 文件：项目库/{project}/03-执行/会议纪要/{日期}-{类型}.md

📋 决议项：{N} 项
📋 待跟踪问题：{M} 项

{如已更新}
✓ 已更新：风险登记册
✓ 已添加：个人工作台待办

{如涉及变更}
⚠️ 识别变更需求，建议走 /change
```

## 与其他 Skill 的协作

| 场景 | 组合 |
|------|------|
| 会议产生变更 | `/meeting` → `/change` |
| 会议识别风险 | `/meeting` → 更新风险登记册 |
| 决议创建专项工作 | `/meeting` → `/work-item action=new source=meeting` |
| 跟踪决议项闭环 | `/meeting` → `/monitor` |
| 启动会后排计划 | `/meeting`（kickoff）→ `/plan` |

---

## Notes

- `/meeting` 负责执行阶段项目会议，售前技术交流请用 `/presales action=tech-exchange`
- 决议项必须有责任人 + 截止日期，否则无法跟踪
- 会议纪要的"风险/变更识别"section 是连接 `/monitor` 和 `/change` 的桥梁
- 建议每次会议都检查上次会议的决议项闭环情况
